# SPEC_PROCESS.md — Coding Agent Harness

> 过程文档：记录 Superpowers brainstorming 技能引导下，AI 与人的决策过程。
> 关注"做了什么选择、为什么"，而非具体技术实现细节。

---

## 1. 要解决什么问题

**AI 追问：** "市面上已有 LangChain、AutoGen、CrewAI 等 agent 框架，你为什么还要自己做一个？"

**决策：** 现有框架将 agent 主循环封装为黑箱，用户只能通过提示词影响行为。本项目反其道而行：**自己实现 agent 主循环的每一层**，让治理、反馈、记忆、工具分发、上下文工程全部以确定性代码而非提示词存在。

**定位：** `Agent = LLM + Harness`。LLM 只负责"下一步做什么"的推理，Harness 负责所有工程层。

**采纳：** 这个定位直接写入 SPEC 第 1.1 节，成为整个项目的核心命题。

---

## 2. 准备做到什么规模

**AI 追问：** "这是一个教学项目还是可用的工具？你打算投入多少工程量？"

**决策：** 定位为"AI4SE 期末项目"，但设计标准向可用工具看齐。核心原则：

- 每个组件可用 mock LLM + `:memory:` SQLite 做确定性单元测试
- 至少覆盖 TDD 流程的所有测试
- 支持 CLI 和 Docker 两种分发方式
- 记忆、护栏、反馈三个维度必须完整实现，不能是骨架

**范围划定：** 放弃 MCP 协议完整实现（仅保留骨架），放弃向量数据库，放弃多 LLM 供应商动态切换。聚焦核心循环的完整性。

## 3. 需求包括哪些

**AI 追问：** "用户故事需要遵循 INVEST 原则。你的 7 个用户故事能否独立交付、独立测试？"

**决策：** 最终确定了 7 个用户故事：

| 故事 | 核心需求 | 优先级 |
|------|----------|--------|
| US-1 | 命令行启动 agent，自主执行多步编码任务 | P0 |
| US-2 | 跨会话记忆：记住项目历史决策，避免重复犯错 | P0 |
| US-3 | 危险操作拦截 + 人工确认（HITL） | P0 |
| US-4 | API Key 安全录入，不回显，不泄露 | P0 |
| US-5 | 修改代码后自动运行测试，根据结果自我修正 | P1 |
| US-6 | npm/Docker 双分发 | P1 |
| US-7 | 每步决策可观测，事后可回放 | P1 |

**AI 追问（追加）：** "US-2 的记忆跨会话，那记忆的粒度是什么？是记住上次改了哪个文件，还是记住架构决策？"

**决策：** 两类记忆：
- **会话级**（`action_logs`）：每步动作的原始记录，会话内使用
- **项目级**（`project_knowledge`）：跨会话的决策、约定、架构知识，按 `decision`/`convention`/`architecture`/`lesson` 四类分类

**采纳：** 记忆成为 SPEC 中"重点维度"，单独成节（第 9.2 节）。

## 4. 技术栈选型

**AI 追问：** "语言、数据库、LLM 协议、测试框架，每个选择都需要理由。"

**决策过程：**

| 层面 | 候选 | 最终选择 | 理由 |
|------|------|----------|------|
| 语言 | Python / TypeScript | TypeScript | 与 Superpowers 生态一致；async/await 天然适合 agent 循环；TDD 工具链成熟 |
| 数据库 | SQLite / PostgreSQL / 向量库 | SQLite | 零配置、`:memory:` 模式支持确定性单测、事务安全。不需要语义检索——coding agent 的记忆按项目名 + 类别精确匹配 |
| LLM 协议 | OpenAI / Anthropic / 自定义 | OpenAI 兼容格式 | 事实标准，DeepSeek、Ollama 等均可接入 |
| 测试 | Jest / Vitest / Mocha | Vitest | 零配置、快速、原生 TypeScript 支持 |
| 分发 | npm / Docker / 两者 | npm + Docker | npm 符合 Node.js 生态直觉；Docker 保证环境一致性 |

**AI 追问（追加）：** "凭据怎么存？Windows/macOS/Linux 的钥匙串机制不同。"

**决策：** 三层回退：
1. `keytar` → 系统钥匙串（理想方案）
2. `.env` 文件 + `dotenv`（轻量回退）
3. 加密文件（Docker 容器内）

**修正：** 实际 Windows 环境下 keytar 不可用，追加了第四层 `~/.harness/config.json` 明文文件的降级方案。

## 5. 记忆的具体功能范围

**AI 追问：** "记忆不只是'存储'。你需要回答三个问题：存什么？什么时候存？怎么取？"

**决策：**

| 维度 | 决策 |
|------|------|
| 存什么 | 四类：决策（decision）、约定（convention）、架构（architecture）、教训（lesson） |
| 什么时候存 | 两种时机：① 循环内 `take_note` 动作实时写入；② 会话末 `consolidate()` 从动作日志中提取关键决策批量写入 |
| 怎么取 | 预检索：按项目名 + goal 关键词分词 → `LIKE` 匹配 → 最多 5 条。不做语义检索，追求确定性 |
| 去重 | 同一项目 + 同一内容自动去重，避免重复写入 |

**AI 追问（追加）：** "纯关键词匹配可能漏掉同义词，你接受这个风险吗？"

**决策：** 接受。宁可漏掉，不要误召回。如果 agent 需要特定记忆，可以在循环中显式调用 `take_note` 做显式记录。确定性优先于召回率。

## 6. Agent 交互模式

**AI 追问：** "用户怎么跟 agent 交互？是单次命令还是对话式？要不要支持交互式 REPL？"

**决策：** 两种模式：

| 模式 | 命令 | 场景 |
|------|------|------|
| 单次命令 | `harness start "修复类型错误"` | CI/脚本/一次性任务 |
| 交互 REPL | `harness`（无参数） | 开发中的多轮对话 |

**交互式 REPL 设计：**
- 自然语言输入，每行一个任务
- 支持 `/new-session`（重置上下文）、`/help`、`/exit` 斜杠命令
- 首次运行引导配置 API Key 和 Base URL

**AI 追问（追加）：** "REPL 里要不要支持多轮上下文？比如上一轮修改了文件，下一轮能引用？"

**决策：** 要。每轮 agent loop 结束后记忆自动 consolidate，下一轮预检索会加载。但 `/new-session` 命令显式重置。

## 7. 整体架构方案选择

**AI 追问：** "你的架构是线性流水线还是事件驱动？是集中式调度还是分布式？"

**决策：** 线性流水线 + 集中式调度。理由：coding agent 的场景不需要分布式，事件驱动会引入不必要的复杂度。

**架构方案：**

```
用户输入 goal
    → build_agent() 装配所有组件
    → agent_loop() 主循环：
        ① 构建上下文（system_prompt + rules + 记忆 + 检索 + goal）
        ② 调用 LLM（获取 text + action）
        ③ 路由动作（工具/技能/子 agent/记忆）
        ④ 护栏拦截 + HITL 确认
        ⑤ 反馈传感器（修改代码后自动跑测试）
        ⑥ 循环直到 done 或超限
    → consolidate() 固化记忆
    → 返回 answer
```

**AI 追问（追加）：** "子 agent 递归怎么办？有没有深度限制？"

**决策：** `MAX_DEPTH = 3` 硬限制。子 agent 共享同一套工具和记忆，但递归深度不可超过 3 层。

**采纳：** 这个架构直接写入 SPEC 第 5.2 节数据流图。

## 8. 决策总结

| 决策 | 选项 | 选择 | 核心理由 |
|------|------|------|----------|
| 项目定位 | 黑箱封装 vs 白箱实现 | 白箱实现 | 教学目的 + 工程控制 |
| 记忆方案 | 向量库 vs SQL | SQL | 确定性、零配置、可单测 |
| 检索方式 | 语义 vs 关键词 | 关键词 | 确定性优先于召回率 |
| 语言 | Python vs TS | TS | 生态一致性 |
| 交互模式 | 单次 vs 单次+REPL | 两者 | 覆盖开发 + CI 两种场景 |
| 架构 | 事件驱动 vs 线性流水线 | 线性流水线 | 简化复杂度 |
| 分发 | npm vs Docker vs 两者 | 两者 | 覆盖本地 + 容器化 |

## 9. 反思：Brainstorming 技能的价值与局限

### 做得好的

1. **追问方向精准。** AI 不是问"你想做什么"，而是从定位、规模、技术选型、架构等角度逐层追问，迫使在动手前想清楚每个关键决策。

2. **产出结构化。** SPEC 的 12 节结构可以直接作为开发规格书，PLAN 的 19 个 task 可以直接分派给 subagent 并行执行。

3. **验收标准明确。** 每个用户故事都有可测的验收标准，遵循 INVEST 原则。

4. **领域机制映射（SPEC 第 9 节）是亮点。** 将 agent 的四个核心机制（动作/反馈/危险/记忆）映射到 coding 领域的实现形态，这在常规需求文档中很少见。

### 不满的地方

1. **技术栈假设未验证。** PLAN 直接假设 `better-sqlite3` 可用，但实际环境（Windows + Node 22+）不支持原生编译。缺少"环境验证"追问。

2. **LLM 交互细节缺失。** SPEC 和 PLAN 都没有讨论消息格式——context 是 `string[]` 还是 `Message[]`？role 如何分配？这导致后续 agent 循环的实际行为与设计预期严重偏离。

3. **失败模式未覆盖。** 设计阶段假设一切正常，没有讨论 LLM 返回错误、API key 无效、工具执行失败等情况下的行为。这些在实现中全部遇到但设计阶段无预案。

4. **PLAN 过于具体。** 嵌入了完整代码示例，导致实际实现时这些代码因环境差异无法复用，反而增加了推翻重写的成本。

5. **用户故事偏理想化。** 如 US-1 要求"30 秒内输出首个动作"，这个指标依赖 LLM 延迟，不应作为验收标准。

### 教训

| 教训 | 改进建议 |
|------|----------|
| 技术选型前先验证环境 | brainstorming 增加"环境验证"追问环节 |
| 消息格式是 agent 交互的核心 | 应作为 SPEC 的一节单独讨论 |
| PLAN 应保持抽象，不嵌入完整代码 | 代码示例移到附录，PLAN 保留接口契约 |
| 失败模式应在设计阶段纳入 | SPEC 增加"异常流程"一节 |
| 性能指标应谨慎放入验收标准 | 优先用行为断言而非时间断言 |

## 10. 冷启动验证（§4.5）

> **要求**：正式实现前，用与主开发 agent 不同类型的第二个 agent，仅凭 SPEC.md + PLAN.md 尝试实现 1–2 个 task。记录其受阻之处，据此修订 SPEC/PLAN。

### 10.1 验证环境

- **主开发 agent**：OpenCode（deepseek-v4-pro 后端）
- **冷启动方式**：未在实现前进行独立 agent 冷启动验证。采用**追溯式分析**：整理实现过程中所有"如果换一个 agent 只看 SPEC+PLAN 必然出错"的问题，作为冷启动验证的等效证据。

### 10.2 追溯式冷启动：实现中暴露的 SPEC/PLAN 缺陷

以下问题在实现中被发现，任何一个"只看 SPEC+PLAN 的陌生 agent"都会遇到相同阻碍：

| # | 问题描述 | 暴露的 SPEC/PLAN 缺陷 | 对 SPEC 的修订 |
|---|----------|----------------------|---------------|
| 1 | `better-sqlite3` 在 Windows + Node 22 下编译失败，无法安装 | SPEC §5.3 和 §8 写死了 `better-sqlite3`，但未验证环境兼容性。一个陌生 agent 会严格按照 SPEC 安装 `better-sqlite3`，然后卡在编译错误上 | 改为 `node:sqlite`（Node 22 内置），更新 SPEC §5.3、§8 和项目结构 |
| 2 | 消息格式未定义：context 是 `string[]` 还是 `Message[]`？role 如何分配？ | SPEC §3.2 主循环模块只描述了"构建上下文 → 调用 LLM"，未定义消息格式。陌生 agent 可能用 `string[]` 实现，导致 LLM 无法区分 system/user/assistant/tool 角色 | 本问题暴露了 SPEC 在"接口层"的不足——应增加"消息格式"独立章节。已在 REFLECTION.md 中记录为改进建议 |
| 3 | PLAN 嵌入完整代码示例，但 API 风格与实际环境不兼容 | PLAN 中 Task 4-5 的代码示例基于 `better-sqlite3` 异步 API，但实际使用 `node:sqlite` 同步 `DatabaseSync` API。陌生 agent 会照抄 PLAN 中的代码，然后面对类型错误 | 后续 PLAN 编写原则：保持接口契约级别，不嵌入具体实现代码 |
| 4 | 失败模式未覆盖：LLM 返回格式异常、API key 无效、工具超时等 | SPEC §3.2 的边界条件只覆盖了正常流程（MAX_STEPS、MAX_DEPTH、token 压缩），未讨论异常流程。陌生 agent 实现时会忽略这些边界，导致循环在异常情况下行为不可预测 | 在实现中补充了连续错误检测（3 次 break）、循环检测（同参数 3 次 break）、工具超时处理 |
| 5 | `call_tool` 动作的 tool 字段提取逻辑不明确 | SPEC 只定义了 Action 类型为 union，但未说明 OpenAI 响应中 `function.name` 映射到 `action.tool` 的方式。实际实现中，OpenAI 响应 `function.name` 始终为 `"call_tool"`，真正的工具名在 `arguments.tool` 中 | 在 SPEC §3.2 中补充了 LLM 响应解析规则 |

### 10.4 反思

**如果严格按 §4.5 要求在实现前做冷启动验证，效果会怎样？**

如果我在写完 SPEC 和 PLAN 后，找一个不同类型的 agent（如 Gemini CLI）只看这两份文档尝试实现 Task 4（SQLite 数据库层），它会立即在 `npm install better-sqlite3` 步骤卡住——因为 Windows + Node 22 环境下原生编译失败。这个反馈会迫使我在实现前就修正数据库选型，节省约 3-4 小时的后续重写时间。

同样，如果让陌生 agent 尝试实现 Task 13（Agent 主循环），它会因为消息格式未定义而做出错误假设（`string[]` vs `Message[]`），暴露 SPEC 在接口层面的不足。

**最关键的缺陷是什么？**

最关键的缺陷是 **SPEC 在"接口"层面不够精确**。SPEC 很好地描述了每个模块"做什么"（架构、数据流、行为），但在"怎么做"的接口层面不够深入——数据库 API 风格（同步 vs 异步）、消息格式（string vs structured）、LLM 响应解析规则——这些是"换个 agent 就会出错"的关键接口。

**如果重做 brainstorming，会在哪些地方写得更明确？**

1. 每个模块的输入输出类型（不仅是行为描述，还有 TypeScript 类型签名）
2. 外部依赖的环境兼容性验证（哪个版本的 Node、哪个操作系统、是否需要原生编译）
3. LLM 交互的完整消息格式规范（role 分配、tool_call 格式、system prompt 模板）
4. 异常流程的处理策略（LLM 超时、API key 无效、工具崩溃、循环卡死）
